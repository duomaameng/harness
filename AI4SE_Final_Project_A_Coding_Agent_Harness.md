## 一、项目概述

本期末项目要求学生综合运用本课程所学的 AI4SE 知识，使用 **[Superpowers](https://github.com/obra/superpowers)** —— 一套面向编码智能体的、内置完整软件开发方法论的技能框架 —— 完成一个有工程深度的真实软件项目，并对其工程化全过程负责。

本项目关注三个核心问题：你能否引导 brainstorming 产出一份足够清晰的 spec？你能否驾驭 plan + subagent 工作流，让智能体长时间自主推进而不脱轨？你能否对智能体生成的代码做出有意义的评审与修正？

课程的命题是：**当 LLM 能完成大部分编码工作时，一个工程师的真正价值在哪里**。Superpowers 替你守住了 TDD、评审、计划这些容易在 AI 协作中松懈的纪律，但它不能替你回答"做什么"和"做对了吗"。这两个问题，是本项目要训练的判断力。

## 二、学习目标

完成本项目后，学生应能够：

1. 使用 Superpowers 的方法论从模糊想法走到可执行的规约与计划。
2. 设计端到端的智能体工作流，掌握 task 拆分、subagent 派发、并行 worktree 等技巧。
3. 在 AI 协作场景中坚持 TDD 与"先验证再宣称完成"的工程纪律。
4. 通过 prompt / context engineering 提升智能体输出质量。
5. 阅读、评审并修正智能体生成的代码。
6. 完成从需求 → 规约 → 计划 → 实现 → 凭据治理 → 分发的完整工程闭环。
7. 对当前 agentic SE 方法论形成自己的批判性见解。

## 三、通用要求

### 3.1 凭据 / API Key 的安全存储（必做）

凡调用付费 LLM 或其它需鉴权的 API，一定要处理凭据：

- key **绝不硬编码**进源码、**绝不提交**进 Git（含历史）、**绝不写入日志 / 终端 history / 明文配置文件**。
- 至少实现一种**安全存储**：操作系统钥匙串（macOS Keychain / Windows Credential Manager / Linux Secret Service）、密钥管理服务、或带主密码的加密文件。环境变量可作为一种来源，但应通过 `.env` 文件加载、而非命令行 `export`（后者会进入 shell history）；并须说明其明文风险（`.env` 为明文、进程环境可见）。
- 首次运行应能**引导用户安全录入** key（如隐藏输入），并能查看 / 更新 / 清除（查看状态时不得回显明文）。
- 在 SPEC 的安全一节中明确凭据威胁模型与对策。

### 3.2 分发（必做，容器 / 二进制 / 包 任选其一或多种）

你要回答的工程问题是：**别人如何获取你的项目并运行起来，且如何安全配置自己的 key？** 任选一种或多种形态：

- **容器镜像**（Docker / OCI）：单条 `docker build` + 单条 `docker run` 可启动；推送到公开 registry。
- **原生可执行二进制**：单文件可执行；说明目标平台与 CPU 架构、是否签名、首次运行的系统拦截如何处理。
- **包管理器分发**：npm / PyPI / cargo / Homebrew 等，给出安装命令。

分发部分必须在 README 写清：获取方式、运行命令、**key 如何在目标机器上安全配置**、已知限制（平台 / 架构 / 依赖前提）。

### 3.3 技术栈不限语言

不要求用某一种语言。TypeScript / Go / Rust / Python 等任意语言均可，需在 SPEC 中说明选型理由。LLM 调用可接任意供应商，同样在 SPEC 中说明。

### 3.4 规模与深度（深度优先）

本项目**不设硬性行数下限**，评价重心是工程深度而非代码量。一个把核心难点想清楚、凭据与分发都做扎实的精炼项目，胜过一个行数堆得很高、却只是把现成框架做一层简单封装的项目。

- 不接受纯 demo 或玩具级项目：你应能简短地向不了解者说清"它为何有人使用"。
- 至少 3 个以上职责清晰的功能模块、可一键运行的测试。
- 凭据与分发是否经得起"在一台全新机器上从零运行"的检验。
- （Coding Agent Harness 项目另有"主角维度做深"与"机制可单测"的深度要求，见 A 文件。）

### 3.5 独立完成

本项目为**个人项目**，不允许组队。你既是 PM、架构师，也是 reviewer 与最终责任人。

### 3.6 工具链

**强制要求：**

- **必须使用 Superpowers 框架**：任选一种支持的编码智能体（Claude Code、OpenAI Codex CLI/App、Cursor Agent、OpenCode、GitHub Copilot CLI、Gemini CLI 均可），按 Superpowers 文档安装插件。
- 必须**如实遵循 Superpowers 的七步工作流**（见 §4）。允许在合理理由下偏离，但偏离必须在 `AGENT_LOG.md` 中记录与解释。
- **TDD 是硬性要求**，由 `test-driven-development` 技能强制执行：先红、再绿、再重构。不接受"先写实现再补测试"。

**条件要求：**

- 凡涉及前端 / UI，强烈推荐使用 **[Open Design](https://github.com/nexu-io/open-design)** 进行界面开发，并在 SPEC 中说明所选设计系统与 skill。纯 CLI / 纯后端项目可豁免。

**鼓励：** 组合使用多种智能体并比较其表现。

## 四、工作流程与交付要求

> **本部分覆盖 Superpowers 全流程：`brainstorming` → `writing-plans` → `using-git-worktrees` → `subagent-driven-development` / `executing-plans` → `test-driven-development` → `requesting-code-review` → `finishing-a-development-branch`。在 SPEC 与 PLAN 完成并通过冷启动验证之前，禁止编写任何实现代码。**

### 4.1 规约与计划生成

启动你的智能体，让 `brainstorming` 技能介入。它会主动追问"你究竟想做什么"，分块呈现设计供你逐步签字确认。把此过程视为一种**与智能体共同设计**的训练：你不是在"领受指令"，而是在"质询"，任何含糊之处都应要求智能体补足。

签字确认设计后，触发 `writing-plans`，将设计分解为"每步 2–5 分钟、明确文件路径、明确验证步骤"的 task 列表。

### 4.2 交付物 1：`SPEC.md`（设计文档）

由 brainstorming 沉淀，必须包含但不限于：

1. **问题陈述**：要解决什么问题？目标用户是谁？为什么值得做？
2. **用户故事**：至少 5 个，遵循 INVEST 原则。
3. **功能规约**：按模块拆分，每项描述输入 / 行为 / 输出 / 边界条件 / 错误处理。
4. **非功能性需求**：性能、**安全（含凭据威胁模型）**、可用性、可观测性。
5. **系统架构**：组件图、数据流、外部依赖（含 LLM 供应商、外部工具）。
6. **数据模型**：主要实体、字段、关系、约束。
7. **凭据与分发设计**：key 的存储方案与录入 / 更新 / 清除流程；分发形态与目标平台、key 在目标机的安全配置方式。
8. **技术选型与理由**：语言、框架、LLM 供应商、分发与部署平台。若含前端，须说明所选 Open Design 设计系统与 skill。
9. **验收标准**：每个功能"完成"的客观判定标准。
10. **风险与未决问题**：你预见到的、可能让智能体出问题的环节。

> Coding Agent Harness 项目（A）须在 SPEC 中额外加入一节「**领域与机制设计**」，具体见对应项目文件。

### 4.3 交付物 2：`PLAN.md`（实现计划）

由 `writing-plans` 产出的任务列表，要求：

- 每个 task 颗粒度足够细，可由一个 subagent 在一次会话内完成。
- 每个 task 包含：目标、涉及文件、预期实现要点、**验证步骤**（含将要写的失败测试）。
- 显式标出 task 间的依赖与可并行部分（用于 worktree 并行）。

### 4.4 交付物 3：`SPEC_PROCESS.md`（过程文档）

记录与 Superpowers 协作生成 spec 与 plan 的过程：

- **brainstorming 关键节点**：智能体追问了哪些好问题？哪些让你修正了原设想？
- **至少 3 轮关键迭代**的对话节选与你的处理决策。
- 哪些建议是 AI 提出而你采纳的？哪些是你推翻或修正的？为什么？
- 反思：brainstorming 技能在你的项目里做得好的地方与让你不满的地方。

### 4.5 自我验证：用"陌生"智能体冷启动试运行

正式实现前，用**一个与主开发智能体不同**的 agent，在**不向其提供你与主 agent 对话历史**的前提下，仅凭 `SPEC.md` + `PLAN.md` 尝试实现 1–2 个 task（约 1–2 小时）。

**为什么必须换一个 agent：** 你和主 agent 在 brainstorming 阶段会沉淀大量共享的隐性上下文，这会让你严重高估 spec 的清晰度。一个全新的 agent 会在你未明文写下的每个假设处受阻；而这些受阻之处，恰恰是 spec 质量最有价值的反馈信号，是单人项目中最接近"同侪评审"的内部机制。

**操作要求：** 第二个智能体类型必须不同；启动全新 session、不导入任何先前会话或 memory；仅提供 SPEC + PLAN，不补充口头解释；指定它从 PLAN 选 1–2 个 task 自主推进，并明确"遇到不确定之处即暂停询问，而非凭猜测继续"。

**记录到 `SPEC_PROCESS.md`：** 第二个 agent 在哪里暂停并提问、暴露了哪些 spec 缺陷？它做出了哪些与你原意不一致的解读（是 spec 写错还是它读错）？产出与预期差距多大？你据此对 SPEC / PLAN 做了哪些修订（给出修订前后的关键 diff）？

**这是规约工作中最关键的"客观证据"。**

### 4.6 实现工作流

1. **git worktrees 隔离工作区**：每个独立功能 / 大模块开一个 worktree，对应一个 PR。
2. **subagent 驱动开发**：每个 task 派一个新鲜 subagent 完成单一任务。
3. **TDD 强制**：红–绿–重构；先编写失败测试、得到失败（红色）结果，再让 subagent 编写最少代码使其变绿，再重构。没有先于测试写出的实现。
4. **两阶段评审**：每个 task 完成后"先 spec 合规检查 → 再代码质量检查"。Critical issue 必须修复才能进入下一 task。
5. **完成分支**：所有 task 完成后由 `finishing-a-development-branch` 决定 merge / PR / 保留 / 丢弃。

### 4.7 GitHub 仓库要求

- 公开 GitHub 仓库（私有仓需将助教加为协作者）。
- **完整的 commit 历史与 PR 工作流**：拒绝单次 commit 提交全部代码；每个 worktree 对应一个 PR。**仓库内不得出现任何真实凭据**——提交前自查 `.env`、history、配置文件。
- 在 commit message / PR 描述中标注由哪个 subagent 完成、人工修改了哪些部分。
- **PLAN.md 持续更新**：每完成一个 task 即标记完成并附 commit hash。
- 维护 `AGENT_LOG.md`（详见 §4.9）。

### 4.8 测试要求

- 必须有可一键运行的测试命令（`make test` 或等价），覆盖核心功能。
- CI（GitHub Actions）必须配置：每次 push 自动运行测试。若选容器分发，CI 还须构建镜像；若选二进制分发，鼓励在 CI 中产出可下载的构建产物。

> Coding Agent Harness 项目（A）另有"核心机制必须用 mock / stub LLM 写确定性单元测试"与"机制演示"的额外测试要求，见对应项目文件。

### 4.9 `AGENT_LOG.md`

按时间顺序记录关键节点，每条包含：时间戳与 task 编号、触发的 Superpowers 技能、关键 prompt / context 配置、subagent 输出的关键片段或 commit hash、人工干预（修改了什么、为什么）、学到的教训。这份日志是实现工作中最重要的"过程证据"。

### 4.10 分发

按 §3.2 选定的形态交付，并在 README 写清：获取与运行命令、key 在目标机器上的安全配置方式、已知限制（平台 / 架构 / 依赖前提）；CI 中包含相应的构建步骤。

### 4.11 云部署

如做带服务端的项目，可选择部署到 Vercel / Render / Railway / Fly.io / 阿里云 / 腾讯云等（多数有学生免费额度）。要求：提供截止前可访问的公网地址；README 说明部署架构与 CI/CD；注意控制成本，优先免费额度。

## A.1 什么是 Harness

核心等式是 **Agent = LLM + Harness**。LLM 相当于 CPU，只负责"决定下一步做什么"这一行任务决策；其余都是工程：把一个只会产生下一步设想的 LLM，封装成一台能稳定、可靠工作的系统。一个 harness 至少要用代码实现以下几件事：

- **决策封装**：组织上下文、调用 LLM、解析其要执行的动作。
- **动作 / 工具**：让 agent 能真正作用于外部世界（读写文件、调用 API、操作外部系统），并将结果回灌给它。
- **上下文与记忆**：决定向模型提供哪些信息，并在多轮、跨会话之间组织与检索。
- **治理护栏**：在危险动作执行前拦截，必要时暂停以等待人工审批（HITL），并用沙箱 / 边界限制其行动空间。
- **反馈闭环**：让 agent 获得关于"其行为是否正确"的客观信号，并据此自我修正。
- **配置**：让使用者通过声明式规则约束 agent 的行为。

本项目的命题落在这层工程上：当 LLM 能完成大部分"思考"时，工程师的价值落在 harness 这层工程（治理、反馈、上下文、安全、分发）。你将"用一个 harness（Superpowers）去造另一个 harness"，从而对这套方法论形成第一手的批判性理解。

## A.2 为什么是 Coding Agent Harness

Coding Agent Harness 面向软件开发场景：它能读写代码、执行命令、运行测试，并根据测试结果自我修正。harness 项目聚焦 coding，是因为它的每个机制都有最清晰、最可编码、也最难以用提示词规避的形态——这恰好使 §A.4"机制必须是代码"的要求最容易落实。

## A.3 你必须设计的四类机制

在 SPEC 中，你必须明确回答并据此设计这四类机制：

- **动作 / 工具**：agent 能执行哪些操作？（coding 通常是读写文件、执行 shell、运行构建与测试）
- **客观反馈信号**：什么信号能告诉 agent"其行为是否正确"、并可回灌以驱动自我修正？（coding 通常是运行测试 / lint / 类型检查，客观、确定、可回灌）
- **危险动作**：什么操作必须暂停以交人工审批、边界如何设定？（coding 通常是删除数据库、危险 shell 命令、对外发布）
- **记忆**：跨会话需要记住什么、信息如何按需提供给 LLM 而非全量载入？（coding 通常是项目约定、历史决策、代码库知识）

## A.4 实现边界：什么必须自己编码（可以使用 AI 工具）

这是本项目最核心的纪律。你用来开发的工具（Superpowers / 编码智能体）**本身就是一个 harness**——它自带主循环、技能机制、钩子、记忆。本项目要求你交付的，是一个**你自己编码实现的 harness 内核**，而**不是在一个现成 agent 框架之上做配置**。两者必须分清：

**澄清"开发工具"与"交付产物"的界线**：开发过程中，你可以充分使用所选编码智能体（Claude Code / Codex / Cursor / Gemini CLI 等任一）与 Superpowers 的全部能力：子 agent、Skill、hooks、memory、内置工具都是辅助你编写代码的工具，鼓励熟练运用。约束只针对你**交付的 harness 内核**：那部分必须是你自己的代码，不能直接调用宿主智能体 / 框架的 agent loop、Skill 机制、治理钩子来充当你产物自身的功能。简言之，宿主框架可以辅助你构建，但不能代替你所构建的那台机器运转。

**(A) 必须自己实现，不得寄生于现成框架**

- 必须自己实现 **agent 主循环**（组织上下文 → 调用 LLM → 解析动作 → 分发执行 → 回灌结果 → 停机判断）。
- 必须有一个**可注入 mock 的 LLM 抽象层**（可替换为 mock LLM 运行离线测试，也可接入真实供应商）。
- **允许**使用底层零件：LLM 供应商的单次对话补全 API、HTTP 库、向量库、解析库等。
- **不允许**把项目建在现成的 agent 编排框架的高层循环之上（如 LangChain `AgentExecutor`、AutoGen、CrewAI、LlamaIndex agent，或某编码智能体 SDK 自带的 agent runner）。把这些零件组装成"循环 + 治理 + 反馈"，必须由你的代码完成。

**(B) 机制必须是代码，不能是提示词**

领域的反馈信号与危险动作，必须落实为**你编写的、确定性的代码机制**，而不是写在提示词里的一句要求：

- 反馈信号 = 一个你编写的**校验器 / 传感器**（解析产物 → 客观判定 → 将结果回灌给循环），而不是"让 LLM 自行检查"的一句提示。
- 危险动作拦截 = 一个你编写的**护栏**（识别危险动作 → 拦截 → 要求人工确认），而不是"提醒 LLM 注意安全"的一句提示。

**(C) 判定标准：移除真实 LLM 后，机制还能用单测验证吗？**

这是区分"编码了机制"与"只写了提示词"的硬标准，也用于评分：

- harness 的每个核心机制（工具分发、治理拦截、反馈回灌、记忆读写、停机），替换为 mock / stub LLM 后，**仍能用确定性单元测试验证它工作**——它才算你编码实现的机制。
- 如果某个"机制"一旦离开真实 LLM 就无法测试（因为它本质上是一句提示词，依赖 LLM 的智能完成），它**不计入** harness 实现。
- **配置文件、规则文件、技能 / 提示词文件都属于"内容物"，不计入 harness 实现工作量。**

以"防止删除数据库的护栏"为例做对比：

- **提示词版（不算实现）**：在系统提示中写一句"不可以执行删除数据库的命令"。是否删除取决于 LLM 是否遵从，且每次结果可能不同，你无法用确定性测试验证它：拦截逻辑并不在你的代码里。
- **代码版（才算实现）**：一个 `guardrail(action)` 函数，识别到危险命令即返回拦截。测试它无需真实 LLM——直接传入一个构造的动作 `guardrail(Action(command="rm -rf /"))`，断言它被拦截，每次都成立。

这一判据的本质是：**移除 LLM 这一不确定因素后，你的仓库里还剩多少可独立验证的工程？** 剩得多，说明你确实实现了一个 harness；剩得少，说明你只是为一个现成 LLM 编写了若干提示词。

**(D) 基础要完整，重点要深入**

六个维度（决策 / 工具 / 记忆 / 治理 / 反馈 / 配置）都要有**可运行的最低实现**——这是 harness "能运行、能闭环、不出错"的基础，缺一项它就不成立；但请**选择其中一个机制密集的维度深入实现，作为你的主要贡献**，而不是六个维度都浅尝辄止。这与写论文同理：各章节都要具备，但须有一个 main contribution 做到足够深入。

建议作为重点的维度：治理（护栏 / 沙箱 / HITL 状态机 / 范围围栏）、反馈闭环（确定性校验器 + 失败分类 + 多轮自我修正）、扩展（工具分发 / 多 agent 编排）——它们天然由代码构成，深入实现后最能体现工程深度，也最契合上文 (A)(B)(C) 的"必须编码"要求。若以记忆 / 上下文工程为重点，其存储与检索必须自己实现，不得直接接用框架自带的 memory。

## A.5 SPEC 的额外要求

在通用要求 §4.2 的 SPEC 结构之上，须额外加入一节：

- **领域与机制设计**：该领域（coding）的反馈信号、危险动作、所需工具、记忆需求分别是什么？你让哪个维度成为重点，为什么？这些机制将如何编码实现（呼应 §A.4）？

## A.6 测试与机制演示的额外要求

在通用要求 §4.8 之上，须额外满足：

- **harness 核心机制必须有用 mock / stub LLM 驱动的确定性单元测试**，且这些测试不依赖网络与真实 LLM。
- 提交一个**机制演示**：在 mock LLM 下确定性地复现以下行为：① 治理护栏拦截一个危险动作；② 注入一次失败，反馈闭环使 agent 收到反馈并据此改变下一步动作；③ 你重点维度的一个确定性行为（与 §A.4-D 的"主要贡献"对齐）。可以是测试用例或可重复运行的脚本。

## A.7 交付物的额外要求

在通用要求 §五清单之上：

- 源代码须包含**你自己实现的 harness 内核**（主循环、工具分发、治理、反馈等），以及覆盖这些机制的 **mock-LLM 单元测试**。
- 提交 §A.6 的机制演示。
